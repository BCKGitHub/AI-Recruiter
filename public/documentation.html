1 What This App Does — The Big Picture

RecruiterAI is a web application designed to help recruiters and hiring managers run AI-powered job interviews automatically. Instead of a human interviewer having to spend time in every initial screening call, an AI voice agent joins the video meeting, asks smart questions, listens to the candidate's answers, and records everything. Afterwards, another AI reads the transcript and produces a structured scorecard for the recruiter to review.

🎯 The Core Workflow in Plain English

  1. A recruiter fills in a form with the candidate's name, email, Zoom link, job description, and resume.
  2. The app saves this information and dispatches a "bot" into the Zoom meeting.
  3. The bot displays an animated AI Interviewer screen and conducts a real voice conversation with the candidate.
  4. Everything said is transcribed and saved in real time.
  5. After the call, the recruiter clicks "Generate Assessment" and receives an AI-written scorecard with scores out of 10 and a hiring recommendation.

Think of it like this: the recruiter sets up the interview in 2 minutes, goes about their day, and comes back to a complete report about the candidate. No human interviewer needed for the screening round.

4 The Database — Where Data is Stored

All interview data is stored in a single database table called interviews, hosted on Supabase's PostgreSQL database. Think of it as a very powerful spreadsheet where each row is one interview.

The interviews Table

Column Name Type What it stores Example
id UUID A unique identifier automatically generated for every interview 3a9f2c11-...
candidate_name Text The candidate's full name Jane Doe
candidate_email Text The candidate's email address jane@example.com
job_description Text The full job description pasted by the recruiter "We are looking for a Senior React developer..."
resume Text The candidate's resume text pasted by the recruiter "Jane has 5 years experience in..."
zoom_url Text The Zoom (or Meet/Teams) meeting link https://zoom.us/j/123456789
recall_bot_id Text (nullable) The ID of the Recall.ai bot that joined the meeting bot_abc123
status Text Where the interview is in its lifecycle Scheduled / In Progress / Completed
transcript JSONB Array of all messages spoken during the interview with roles and timestamps [{"role":"interviewer","content":"Hello Jane..."}]
assessment JSONB The Claude AI-generated scorecard with scores, rationale, and recommendation {"technicalSkillsMatch": 8, ...}
created_at Timestamp When the interview was created 2026-06-16 10:30:00 UTC

⚠️ What is JSONB?

JSONB is a flexible data type in PostgreSQL that stores structured data as JSON. For example, instead of having 50 separate columns for each transcript message, the entire conversation is stored as one field containing a list of objects. This makes it easy to store data of variable length and complexity.

Security: Row Level Security (RLS)

The database uses a security feature called Row Level Security (RLS). This means rules are set on who is allowed to read and write data. In this app, anonymous (not logged in) users are allowed full access — this is intentional because the app is designed as an internal recruiter tool, not a public consumer app. If user accounts were added later, these policies would be tightened.

5 Frontend Pages & Components

The frontend is a Single Page Application (SPA) — meaning the entire app lives in one HTML file and React swaps different views in and out without navigating to a new page. Think of it like a TV with multiple channels: the screen stays the same, but what's shown changes.

5.1 — App.tsx: The Main Shell

App.tsx is the root component — the outermost wrapper that everything else sits inside. It has two jobs:

  • Route detection: If the URL is /agent, it shows the special AI Interviewer screen. Otherwise, it shows the recruiter dashboard.
  • Navigation state: It tracks which page the recruiter is on (Setup, Interviews, or Transcript/Assessment) and passes instructions to child components.

Sidebar Navigation

The sidebar on the left has two nav buttons: "Setup Interview" and "Active Interviews". Clicking them changes the page state variable, which controls what is shown in the main content area. The RecruiterAI logo and branding sit at the top of the sidebar.

5.2 — SetupInterview.tsx: Create a New Interview

This is the form page recruiters fill in to create a new interview session. It collects five fields:

  • Candidate Name — who is being interviewed
  • Candidate Email — for reference
  • Zoom URL — the meeting link the bot should join
  • Job Description — pasted text of the role requirements
  • Resume — pasted text of the candidate's CV

When the recruiter clicks "Start Interview", two things happen in sequence:

1
Save to Supabase The form data is inserted as a new row in the interviews table with status "Scheduled".
2
Call the Edge Function The app calls create-recall-bot with the interview ID and Zoom URL, which dispatches a bot into the meeting.
3
Show Result A green success or red error message appears confirming whether the bot was created successfully.

5.3 — ActiveInterviews.tsx: View All Interviews

This page shows a table of every interview that has been created. Each row shows the candidate's name, email, date, status badge (Scheduled / In Progress / Completed), and how many transcript messages have been captured. Clicking any row opens the Transcript & Assessment view for that interview.

The data is fetched directly from Supabase every time the page loads, and there is a Refresh button to reload it manually. The interviews are sorted newest first.

5.4 — TranscriptAssessment.tsx: Review & Assess

This is the most information-rich page. It shows:

  • The full interview transcript — displayed as a chat bubble layout, with AI messages on the left and candidate messages on the right. Each bubble shows the timestamp.
  • Generate Assessment button — calls the generate-assessment Edge Function which uses Claude AI to analyse the transcript.
  • The AI scorecard — once generated, shows:
    • An overall recommendation (Strong Yes / Yes / Maybe / No)
    • Score bars for Technical Skills, Communication, and Depth of Experience
    • Detailed score cards with a score out of 10, rationale, and a supporting quote from the transcript
    • Culture fit indicators (bullet list of positive observations)
Tip: The assessment can be regenerated at any time by clicking the Regenerate button — useful if the transcript gets updated or you want a fresh perspective.

5.5 — AgentPage.tsx: The AI Interviewer Screen

This page is not seen by recruiters — it is the screen that the Recall.ai bot displays inside the Zoom meeting to the candidate. It is a full 1280×720px dark-themed page with:

  • An animated robot avatar with a pulsing ring that changes colour based on status
  • A status pill showing "Connecting", "Listening", or "AI Interviewer speaking..."
  • A preview of the last 3 lines of transcript as the conversation unfolds

When this page loads, it reads the interviewId from the URL, fetches the interview data from Supabase, connects to OpenAI's Realtime API over WebRTC, and immediately starts the conversation. The AI speaks first, greeting the candidate by name.

🎙️ How the Voice Works (Simplified)

WebRTC is a browser technology that enables live audio streams. The AgentPage opens a peer-to-peer connection with OpenAI's servers. The bot's microphone audio goes to OpenAI, and OpenAI's voice response comes back as audio and is played through the bot's virtual speaker — which the candidate hears in the Zoom call.

6 Backend Edge Functions

Edge Functions are small programs that run in the cloud — not in the user's browser and not on a traditional server. They run on Supabase's infrastructure using the Deno runtime. They are perfect for tasks that need secret API keys, since those keys never leave the server.

6.1 — openai-realtime-token

URL: POST /functions/v1/openai-realtime-token
Purpose: Issues a short-lived ephemeral key that allows the browser to connect to OpenAI's Realtime API without exposing the main OpenAI API key.

Why is this needed? The real OpenAI API key must never be placed in the browser — anyone visiting the page could extract it and use it to run up charges. This Edge Function holds the real key securely and exchanges it for a temporary "one-use" token that the browser can safely use for one session.

What it does:

  1. Reads OPENAI_API_KEY from secure environment variables.
  2. Calls https://api.openai.com/v1/realtime/client_secrets to generate an ephemeral key.
  3. Returns the token to the browser.
// Session configuration sent to OpenAI
{
  session: {
    type: "realtime",
    model: "gpt-realtime-2",
    audio: {
      output: { voice: "marin" }
    }
  }
}

6.2 — create-recall-bot

URL: POST /functions/v1/create-recall-bot
Purpose: Sends a Recall.ai bot into the specified video meeting, configured to display the AI Interviewer screen and conduct the interview.

What it does:

  1. Receives interviewId and zoomUrl from the frontend.
  2. Constructs the Agent Page URL: https://your-app.netlify.app/agent.html?interviewId=XXX
  3. Calls Recall.ai's API to create a bot that joins the meeting and shows that URL as its camera feed.
  4. Updates the interview record in Supabase with the bot's ID and sets status to "In Progress".

The Recall.ai bot is configured with:

  • output_media.camera — displays the AgentPage HTML as a virtual webcam
  • recording_config — enables transcript capture with speaker diarization (distinguishing who is talking)
  • variant — uses a high-performance 4-core browser engine

6.3 — generate-assessment

URL: POST /functions/v1/generate-assessment
Purpose: Uses Claude AI (claude-opus-4-5) to analyse the interview transcript and produce a structured hiring scorecard.

What it does:

  1. Receives interviewId.
  2. Fetches the interview (including transcript and job description) from Supabase.
  3. Formats the transcript as readable text and sends it to Claude with a detailed prompt.
  4. Parses Claude's JSON response into a structured assessment object.
  5. Saves the assessment back to the interview record in Supabase.

The prompt instructs Claude to return a JSON object with these exact fields:

{
  "technicalSkillsMatch":    7,          // Score 1–10
  "technicalSkillsRationale": "...",     // Explanation
  "technicalSkillsQuote":     "...",     // Exact quote from transcript
  "communicationClarity":     8,
  "communicationRationale":   "...",
  "communicationQuote":       "...",
  "depthOfExperience":        6,
  "depthRationale":           "...",
  "depthQuote":               "...",
  "cultureFitIndicators":     ["...", "...", "..."],
  "overallRecommendation":    "Strong Yes",  // or Yes / Maybe / No
  "recommendationSummary":    "..."
}

6.4 — recall-transcript-webhook

URL: POST /functions/v1/recall-transcript-webhook (also handles /call-end sub-path)
Purpose: Receives incoming transcript chunks and call-end notifications from Recall.ai as the interview is happening.

This Edge Function is a webhook — meaning it is a URL that Recall.ai calls, rather than us calling it. It handles two kinds of events:

  • Transcript chunk: When a participant speaks, Recall.ai sends the words and who said them. The function identifies the speaker (interviewer or candidate), appends the new message to the transcript array in Supabase, and saves it.
  • Call end (/call-end): When the meeting ends, Recall.ai calls this endpoint. The function updates the interview status to "Completed".
Note on speaker identification: The function uses heuristics to determine whether a speaker is the interviewer or the candidate — checking if the speaker name contains "interview" or if they are identified as participant "0". This works for the common case but may need refinement for edge cases.

7 The AI Voice Interview — Step by Step

The most complex part of the system is how the live voice conversation actually works. Here is every step in the exact order it happens during a real interview:

1
Recall.ai bot joins the Zoom meeting The bot appears in the meeting as a participant named "AI Interviewer". Its camera shows the AgentPage HTML rendered in a headless browser (a browser with no visible window).
2
AgentPage loads inside the bot's browser The page reads ?interviewId=XXX from the URL and fetches the interview data (candidate name, resume, job description) from Supabase.
3
Ephemeral token is fetched The page calls the openai-realtime-token Edge Function to get a secure, short-lived key. This key will expire after the session and cannot be reused.
4
WebRTC connection established An RTCPeerConnection is created. The bot's microphone audio track is added to the connection. An SDP offer (a technical description of the audio session) is sent to OpenAI's /v1/realtime/calls endpoint, which responds with the matching answer.
5
Session is configured via data channel A data channel named oai-events is opened. The app sends a session.update event with the system prompt (including the resume and job description), enables candidate speech transcription via audio.input.transcription, and configures voice activity detection (VAD).
6
AI speaks first A response.create event is sent immediately after the session update, instructing the AI to greet the candidate by name and ask its first question.
7
Conversation continues automatically Server-side VAD (Voice Activity Detection) handles turn-taking. When the candidate stops speaking for 700ms, OpenAI processes their input and the AI formulates and speaks its next response. No manual interruption handling is needed.
8
Transcript is saved in real time Two event types populate the transcript: response.audio_transcript.done (interviewer's words) and conversation.item.input_audio_transcription.completed (candidate's words). Each is appended to the transcript array and saved to Supabase with a 1.5 second debounce (to avoid excessive database writes).
9
Interview ends When the meeting ends, Recall.ai sends a webhook to /recall-transcript-webhook/call-end, which marks the interview as "Completed" in Supabase.

The OpenAI Realtime API — Key Technical Details

The app uses gpt-realtime-2, which is OpenAI's latest GA (Generally Available) voice model. This model uses a different session configuration schema to older models like gpt-4o-realtime:

Setting Old model schema (gpt-4o-realtime) New model schema (gpt-realtime-2) — used here
Transcription config input_audio_transcription: { model: "whisper-1" } audio.input.transcription: { model: "whisper-1" }
Voice activity detection turn_detection: { type: "server_vad" } audio.input.turn_detection: { type: "server_vad" }
Output voice voice: "alloy" at session level audio.output.voice: "marin"
SDP endpoint POST /v1/realtime?model=... POST /v1/realtime/calls

8 Deployment & Configuration

Frontend — Netlify

The React app is deployed to Netlify, a hosting service that serves static files. The build process runs npm run build which uses Vite to compile all TypeScript/React code into plain HTML, CSS, and JavaScript files in a dist/ folder.

netlify.toml — The Deployment Configuration File

[build]
  command = "npm run build"
  publish = "dist"

[build.environment]
  VITE_SUPABASE_URL = "https://bvckybdvplsjbjhspbyp.supabase.co"
  VITE_SUPABASE_ANON_KEY = "eyJhbGci..."

[[redirects]]
  from = "/*"
  to = "/index.html"
  status = 200

The [[redirects]] rule is critical — it tells Netlify to always serve index.html regardless of the URL path, which is required for Single Page Applications.

Environment Variables

Variable Where it lives Purpose
VITE_SUPABASE_URL Netlify build environment The URL of the Supabase project (public, safe to expose)
VITE_SUPABASE_ANON_KEY Netlify build environment The anonymous key for reading/writing data (public, safe with RLS)
OPENAI_API_KEY Supabase Edge Function secrets The real OpenAI API key — NEVER in the browser
ANTHROPIC_API_KEY Supabase Edge Function secrets The Anthropic (Claude) API key — NEVER in the browser
RECALL_API_KEY Supabase Edge Function secrets The Recall.ai API key for sending bots to meetings
APP_URL Supabase Edge Function secrets The public Netlify URL, used to build the agent page link for the bot
SUPABASE_SERVICE_ROLE_KEY Supabase Edge Function secrets (auto-injected) A privileged key for Edge Functions to bypass RLS when writing data

The agent.html File

There is a special file at public/agent.html that is a standalone HTML page (not part of the React app). It is served directly by Netlify and loaded by the Recall.ai bot. The Vite build process copies it to dist/agent.html and injects the Supabase URL and key into the file's JavaScript at build time.

9 Key Bugs Fixed During Development

Several important issues were diagnosed and resolved during development. Understanding these helps future developers avoid the same pitfalls.

Bug 1 — Candidate Transcript Not Being Captured

Symptom

Only the AI interviewer's side of the conversation appeared in the transcript. The candidate's spoken words were never saved.

Root Cause

The session.update event sent to OpenAI used the old gpt-4o-realtime schema. The old schema places input_audio_transcription at the top level of the session object. The new gpt-realtime-2 GA API nests this setting inside audio.input.transcription. Sending the wrong structure was silently ignored, so transcription was never enabled for the candidate's audio.

Fix

// WRONG (old schema):
session: { input_audio_transcription: { model: "whisper-1" } }

// CORRECT (gpt-realtime-2):
session: {
  type: "realtime",
  audio: {
    input: {
      transcription: { model: "whisper-1" },
      turn_detection: { type: "server_vad", ... }
    }
  }
}

Bug 2 — Blank White Page on Netlify

Symptom

After deploying to Netlify, the site showed a blank white page. The browser console showed: Uncaught Error: supabaseUrl is required.

Root Cause

Two separate issues: (1) There was no netlify.toml at the project root providing the environment variables at build time. Netlify did not know about VITE_SUPABASE_URL so Vite compiled the app with empty strings. (2) There was no _redirects rule, so refreshing any page returned a 404.

Fix

Created a netlify.toml at the project root with [build.environment] containing both Supabase variables, and a [[redirects]] rule pointing /* to /index.html.

Bug 3 — SDP Negotiation Failure with gpt-realtime-2

Symptom

The WebRTC connection to OpenAI failed with an SDP error.

Root Cause

The old ephemeral-key WebRTC path sent the SDP offer directly to https://api.openai.com/v1/realtime?model=... with Content-Type: application/sdp. The new gpt-realtime-2 GA API requires posting to /v1/realtime/calls with a short-lived ephemeral key as the Bearer token.

Fix

Updated the edge function to call /v1/realtime/client_secrets for token generation, and updated agent.html to post the SDP to /v1/realtime/calls.

10 Glossary for Non-Technical Readers

Term Plain English Explanation
API Application Programming Interface — a way for one software program to talk to another. Like a waiter who takes your order to the kitchen and brings back food.
React A tool for building web pages that update dynamically without refreshing, like a news ticker that changes without reloading the whole page.
TypeScript A stricter version of JavaScript (the language that makes websites interactive) that catches mistakes before the code runs.
Component A reusable building block of a web page — like a LEGO brick. A button, a form, or a whole page section can each be a component.
Database / Supabase A structured place to store and retrieve data — like a very powerful, searchable spreadsheet hosted in the cloud.
Edge Function / Serverless A small program that runs in the cloud only when needed, rather than on a permanently running server. Like a vending machine — available 24/7 but only uses power when someone uses it.
WebRTC A browser technology that enables direct real-time audio/video communication — it's what makes video calls work in a browser without installing software.
SDP (Session Description Protocol) A technical "hello, here is what audio formats I support and my network details" message that two computers exchange to set up a voice call.
Webhook A URL that receives notifications from external services. Instead of constantly asking "are there any updates?", the service sends a message to your webhook when something happens — like a doorbell vs. constantly knocking.
VAD (Voice Activity Detection) Technology that automatically detects when someone is speaking vs. silent, used to know when the candidate has finished talking so the AI can respond.
Ephemeral token / key A temporary password that expires quickly (usually after one use or a short time). Used so the browser can connect to an API without storing a permanent secret key.
JSONB A database column type that stores flexible, structured data (like a list of messages) rather than simple text or numbers.
RLS (Row Level Security) Database security rules that control exactly who can see or change which rows of data.
SPA (Single Page Application) A website that loads once and then swaps content dynamically without full page reloads — like a phone app that shows different screens without restarting.
Netlify A service that hosts websites on the internet. You give it your code files and it makes them publicly accessible via a URL.
Deno A runtime environment for running JavaScript/TypeScript programs on a server (similar to how a browser runs JavaScript, but for the backend).
Recall.ai A service that provides programmable bots that can join video calls (Zoom, Google Meet, Teams) and display custom content or capture audio/video.
Claude AI / Anthropic An artificial intelligence assistant that can understand and generate text — used here to read interview transcripts and write evaluation reports.
OpenAI Realtime API An OpenAI service that enables real-time two-way voice conversations with an AI model — similar to talking to a person, but the "person" is an AI.
Deployment The process of taking code from a developer's computer and making it live on the internet for people to use.
Build The process of compiling and packaging source code into optimised files that browsers can efficiently load and run.
